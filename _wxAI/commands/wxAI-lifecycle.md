## Lifecycle Tracker

This command generates or updates `project-documentation/Lifecycle.md` -- a master project tracking document that maps all wxAI pipeline phases, and task completion into a single view.

**Output**: `project-documentation/Lifecycle.md`

---

## Pre-Flight

1. **Parse arguments** for flags:
   - `--audit-only`: Print audit report to console without writing Lifecycle.md
   - `--force`: Regenerate from scratch even if Lifecycle.md exists
   - (no flags): Update existing Lifecycle.md or create if missing

2. **Validate environment**:
   - Confirm `specs/` directory exists
   - Confirm `project-documentation/` directory exists (create if needed)

---

## Data Gathering

### Step 1: Scan Spec Directories

List all directories under `specs/` matching the `NNN-*` pattern. For each spec directory, check for the presence of these artifacts to determine pipeline phase completion:

| Artifact                     | Phase Indicated                    |
| ---------------------------- | ---------------------------------- |
| `spec.md`                    | Specify phase complete             |
| `checklists/requirements.md` | Clarify phase complete             |
| `plan.md`                    | Plan phase complete                |
| `research.md`                | Plan phase complete (sub-artifact) |
| `data-model.md`              | Plan phase complete (sub-artifact) |
| `contracts/`                 | Plan phase complete (sub-artifact) |
| `quickstart.md`              | Plan phase complete (sub-artifact) |
| `tasks.md`                   | Tasks phase complete               |

**Implementation status** is determined by:

- If tasks.md exists: Parse `- [x]` (completed) vs `- [ ]` (pending) lines
  - 100% checked = "Done"
  - > 0% checked = "In Progress (NN%)"
  - 0% checked = "Not Started"
- If tasks.md does not exist: Mark as "Not Started"

### Step 4: Parse Task Progress

For each spec that has a `tasks.md` file:

1. Count lines matching `- [ ]` (unchecked tasks)
2. Count lines matching `- [x]` or `- [X]` (checked tasks)
3. Calculate completion percentage: `checked / (checked + unchecked) * 100`

## Document Generation

Generate `project-documentation/Lifecycle.md` with the following structure:

### Header

```markdown
# HotshotTrucking Project Lifecycle

> Master tracking document for all project phases, specifications, and work items.
> **Last Updated**: YYYY-MM-DD | **Generated By**: `/wxAI-lifecycle`

**Totals**: N specs | T total tasks
```

### Summary Dashboard

Create a table showing all specs with their pipeline phase status:

```markdown
## Summary Dashboard

| Spec | Name              | Specify | Clarify | Plan | Tasks | Implement |
| ---- | ----------------- | ------- | ------- | ---- | ----- | --------- |
| 001  | using-docs-design | Done    | -       | Done | Done  | Done      |
| ...  | ...               | ...     | ...     | ...  | ...   | ...       |
```

Use checkmark or "Done" for completed phases, dash for skipped/not applicable, "In Progress" for active work.

### Active Spec Sections

For each spec that is not fully implemented, create a detailed section:

```markdown
## Spec NNN: Feature Name

### Pipeline Status

- [x] Specify: `specs/NNN-name/spec.md`
- [x] Clarify: `specs/NNN-name/checklists/requirements.md`
- [x] Plan: `specs/NNN-name/plan.md`
- [x] Tasks: `specs/NNN-name/tasks.md`
- [ ] Implement: In Progress (NN% - XX/YY tasks)

```

### Completed Specs

For specs 001-005 (fully implemented), show a collapsed summary:

```markdown
## Completed Specs

<details>
<summary>Specs 001-005 (all phases complete)</summary>

| Spec | Name              | All Phases | Notes             |
| ---- | ----------------- | ---------- | ----------------- |
| 001  | using-docs-design | Done       | Foundation design |
| ...  | ...               | ...        | ...               |

</details>
```

### Audit Notes

Include a section documenting:

- Any features found in code without corresponding tracking in `tasks.md`.
- Recommendations for cleanup

```markdown
## Audit Notes

### Missing Tracking

- Feature X exists in code but has no task: `path/to/code`
```

### Footer

```markdown
---

_Generated by `/wxAI-lifecycle` on YYYY-MM-DD. Re-run `/wxAI-lifecycle` to update._
```

## Post-Generation

1. **If `--audit-only` was set**: Print the audit findings to console and stop. Do not write Lifecycle.md.

2. **Otherwise**: Write the generated document to `project-documentation/Lifecycle.md`.

3. **Open the file in the editor**: After writing, open the file so the user can see it immediately.

4. **Print summary**:
   ```
   ===================================================
   LIFECYCLE TRACKER UPDATED
   ===================================================
   File:           project-documentation/Lifecycle.md
   Specs Tracked:  N
   Task Progress:  X/Y total tasks complete
   Audit Issues:   Z findings
   ===================================================
   ```

---

## Error Handling

- If `specs/` directory does not exist: Report error and suggest running `/wxAI-pipeline` to create the first spec.
- If a tasks.md file has unexpected format: Skip task counting for that spec, note in audit.
- If `project-documentation/` does not exist: Create the directory.

---

## Behavior Rules

- **Idempotent**: Running the command multiple times produces the same output given the same source data.
- **Non-destructive**: The command only writes to `project-documentation/Lifecycle.md`. It never modifies spec files or tasks.md files.
- **Source of truth**: The Lifecycle.md is a **derived document** -- the actual sources of truth are the spec artifacts in `specs/`. If Lifecycle.md becomes stale, simply re-run the command.
- **Absolute paths**: All file paths in the generated document should be relative to the repo root for portability.
